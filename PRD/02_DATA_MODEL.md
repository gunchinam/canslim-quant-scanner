# 판단 포스터 한줄평 헤드라인 레이아웃 재설계 -- 데이터 모델

> 이번 작업은 순수 레이아웃/CSS/JS 재설계로, 새로운 데이터 엔티티를 추가하지 않는다.
> 기존에 이미 API 응답에 존재하는 필드를 그대로 사용한다.

---

## 전체 구조

```
[종목 API 응답(d)] --1:1--> [OneLiner 필드] --렌더링--> [판단 포스터 헤드라인]
                     |
                     └--1:1--> [conv/EntryScore 등] --계산--> [_pgCls, 확신도%, _pvReason]
```

---

## 사용하는 기존 필드

### `d.OneLiner`
`web_app/app.py`의 `_annotate_one_liners` -> `one_liner.annotate()`가 채우는 필드. `one_liner.py`의 `_PHRASES`(18개 버킷, 3533개)에서 티커+지표 해시로 1개를 선택한 문자열.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| OneLiner | 종목 성격(버킷)에 대한 한줄평 문장, 15~40자 | "이거 매수하면 주갤에서 박제당함" | O |

### `conv` 및 관련 지표 (변경 없음, 참고용)
`app.js`의 `_renderEntryVerdict(d)` 내부에서 `BFScore`/`EntryScore`/`TotalScore`/`GreedZone`/`MDD`로 계산되는 진입 타이밍 확신도(0-100). 이번 골에서 계산 로직은 건드리지 않는다.

### 관계
- 종목 1개는 `OneLiner` 1개를 가진다(같은 티커도 갱신 주기에 따라 값이 바뀔 수 있음).
- `OneLiner`(텍스트)와 `conv`(진입 타이밍 점수)는 서로 다른 축의 정보이며, 이번 골에서도 이 분리는 유지된다 -- 텍스트는 OneLiner, 배경색/확신도%는 conv.

---

## 왜 이 구조인가

- 단순성: 새 데이터를 추가하지 않고 기존 `OneLiner` 필드의 "표시 방식"만 바꾼다. 백엔드(`app.py`, `one_liner.py`) 변경이 필요 없다.
- 일관성: 직전 골에서 확정한 "텍스트=OneLiner, 색상/확신도=conv" 분리 원칙을 그대로 유지해 두 축이 다시 뒤섞이지 않게 한다.
