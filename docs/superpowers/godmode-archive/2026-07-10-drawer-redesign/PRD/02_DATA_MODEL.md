# 종목 상세 드로워 재설계 — 데이터 모델

> 이 작업은 새 데이터 필드나 API를 추가하지 않는 순수 UI 재배치입니다.
> 아래는 "새 엔티티"가 아니라, 드로워 안에서 기존 데이터를 어떻게 그룹핑해서 보여줄지에 대한 정리입니다.

---

## 전체 구조 (정보 그룹)

```
[결론군] --근거로--> [근거군] --보조 설명--> [보조군]
                                                  |
                                                  v
                                             [탭군] (상세 지표)
```

- 결론군: 사용자가 드로워를 열자마자 봐야 하는 값.
- 근거군: 결론을 뒷받침하는 가격/밸류 지표.
- 보조군: 결론·근거를 보완하는 요약 카드.
- 탭군: 더 깊이 볼 때만 진입하는 상세 지표.

---

## 그룹 상세

### 결론군
드로워를 열자마자 가장 먼저 눈에 들어와야 하는 값. 기존 API 응답(`detail-score`, `detail-signal`, `detail-oneliner` 등)을 그대로 사용.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| detail-score | 종합 점수(재무·밸류·기술 합산) | 78 | O |
| detail-signal | 등급/판정(예: 매수, 관망 등) | 매수 | O |
| detail-sector | 섹터 배지 | 반도체 | X |
| detail-oneliner | 한줄평(범위 제외 컴포넌트, hero-oneliner) | — | X |

### 근거군
결론을 뒷받침하는 가격·밸류 지표. 히어로 카드 하단에 위치.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| detail-price | 현재가 | 72,300 | O |
| detail-day-chg | 전일 대비 등락 | +1.2% | X |
| detail-rsi | RSI 지표 | 54.2 | X |
| detail-target | DCF 적정가 | 85,000 | X |
| detail-broker-target | 증권사 목표가 | 90,000 | X |
| cons-low / cons-mean / cons-high | 컨센서스 최저/평균/최고 | 70,000 / 85,000 / 95,000 | X |

### 보조군
결론·근거를 보완하는 요약 카드. 데이터가 없으면 카드 자체가 숨겨짐(`display:none`).

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| detail-earnings-chips | 실적 한눈에 칩 목록 | 매출 YoY +12% | X |
| dp-investor-grid | 투자자 동향(기관/외국인 등) 그리드 | 기관 순매수 3일 | X |

### 탭군
탭을 클릭해야만 로딩되는 상세 지표. 각 탭의 내부 데이터 구조는 이번 작업에서 변경하지 않음.

| 탭 | 설명 | 필수 |
|------|------|------|
| CAN SLIM | CAN SLIM 항목별 브레이크다운 리스트 | O(항상 노출) |
| 기술 지표 | 기술적 분석 지표 | O(항상 노출) |
| 재무 지표 | 재무제표 기반 지표 | O(항상 노출) |
| 공시·뉴스 | 공시/뉴스 피드 | X(데이터 있을 때만 노출) |
| US 인사이트 | 미국 시장 관련 인사이트 | X(데이터 있을 때만 노출) |
| 노무라식 | 노무라식 분석 | X(데이터 있을 때만 노출) |

### 관계
- 결론군 1개(현재 보고 있는 종목)에 근거군·보조군이 종속됨(1:1, 종목 단위).
- 탭군은 결론군과 독립적으로 지연 로딩되며, 종목당 최대 6개 탭 중 데이터 존재 여부에 따라 3~6개가 노출됨.

---

## 왜 이 구조인가

- **확장성**: 이번 재배치는 CSS/마크업 그룹핑만 바꾸므로, 추후 새 지표가 추가되어도 "근거군에 필드 하나 추가" 식으로 확장 가능.
- **단순성**: 새 데이터 계층을 만들지 않고 기존 DOM id(`detail-score`, `detail-price` 등)를 그대로 재사용해 백엔드/JS 로직 변경 리스크가 없음.
