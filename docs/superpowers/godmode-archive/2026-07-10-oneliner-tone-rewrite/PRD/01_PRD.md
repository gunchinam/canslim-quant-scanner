# 한줄평 엔진 톤 통일 리라이트 — PRD (Product Requirements Document)

> 생성일: 2026-07-10
> 생성 도구: Show Me The PRD

---

## 1. 제품 개요

### 한 줄 요약
종목 스캐너 웹앱의 한줄평 생성 엔진(`web_app/one_liner.py`)에서, 여러 세대에 걸쳐 누적된 문구 풀(base/V1/V2/V3/V4/밈, 총 3449개)을 spicy_v2 특유의 날것 주갤밈 슬랭 톤으로 통일 리라이트해 엔진의 정체성을 회복한다.

### 해결하는 문제
`one_liner.py`는 18개 버킷 × 여러 세대의 문구 풀이 `_PHRASES` 딕셔너리에 누적 병합되어 있다. spicy_v2(4개 모듈, 856개, "ㄹㅇ/ㅇㅈ/존버/줍줍/깡통/호구" 같은 날것 슬랭)는 코드에서 죽지 않고 여전히 쓰이고 있지만, 전체의 24.8%에 불과해 나머지 75.2%(base 1778개 + V1 270개 + V3 270개 + V4 179개 + 밈 96개, 상대적으로 순화된 밈 톤)에 확률적으로 묻힌다. 그 결과 사용자가 실제 화면에서 느끼는 전체 인상이 "엔진의 정체성이 애매함"으로 나타난다.

### 핵심 가치
- 문구 풀 구조(`_PHRASES: dict[버킷명 -> list[문구]]`)나 버킷 분류 로직(`_bucket()`, `_raw_bucket()`)은 건드리지 않는다 — 이미 검증된 골격을 유지.
- V2를 지우거나 비중만 조절하는 게 아니라, 전체 문구를 V2와 같은 톤으로 다시 쓴다 — "여러 톤이 섞인 상태"가 아니라 "하나의 색깔"로 수렴시키는 것이 이번 작업의 차별점.

## 2. 사용자

### 주요 사용자
- **누구**: 종목 스캐너 앱을 실사용하는 개인 투자자(제작자 본인 포함, 소수 사용자) — 드로워 재설계 PRD와 동일한 사용자.
- **상황**: 스캐너 목록/드로워에서 한줄평을 읽으며 종목에 대한 첫인상을 형성하는 상황.
- **목표**: 한줄평을 읽었을 때 "이 엔진 특유의 목소리"를 느끼고 싶다 — 밋밋한 설명문이 아니라 주갤 커뮤니티 감성이 살아있는 한마디.

### 사용자 시나리오
1. 사용자가 스캐너 목록/드로워에서 종목 카드를 보면
2. 한줄평이 표시되고
3. 그 문구가 버킷(부정/긍정/중립)과 무관하게 일관된 날것 슬랭 톤("ㄹㅇ", "줍줍", "떡상" 류)으로 느껴진다

## 3. 핵심 기능

| 기능 | 설명 | 우선순위 | 복잡도 |
|------|------|----------|--------|
| negative 그룹 리라이트 | VALUE_TRAP, BUBBLE, OVERBOUGHT, FALLING_KNIFE, AVOID 버킷의 base/V1/V3/V4/밈 문구를 V2 톤으로 재작성 | P1 (MVP) | 보통 |
| positive_a 그룹 리라이트 | TRUE_VALUE, EXPENSIVE_JUSTIFIED, MOMENTUM_LEADER, BREAKOUT 버킷 재작성 | P1 (MVP) | 보통 |
| positive_b 그룹 리라이트 | SLEEPING_GIANT, CASH_COW, SECTOR_LEADER, DEFENSIVE 버킷 재작성 | P1 (MVP) | 보통 |
| mixed 그룹 리라이트 | EARNINGS_BEAT, STORY_STOCK, OVERSOLD, NEUTRAL 버킷 재작성 | P1 (MVP) | 보통 |
| STRONG_BUY 리라이트 + 보강 | 기존 111개 재작성 + 신규 84개 추가로 195개까지 확충 | P1 (MVP) | 간단 |
| 자동 검증 게이트 | 버킷마다 기존 테스트 규칙(금지어/중복/격식체/NEUTRAL 방향성 등)을 통과할 때까지 재생성 | P1 (MVP) | 간단 |
| 무작위 표본 육안 검수 | 버킷 완료 후 5~10개 무작위 표본을 사람이 읽고 톤 확인 | P1 (MVP) | 보통 |

## 4. 사용자 흐름 (User Flow)

### 핵심 흐름
```
버킷 그룹 선정 -> 그룹 내 각 버킷 문구 리라이트 -> 자동 검증(반복) -> 표본 육안 검수 -> 다음 그룹
```

### 상세 흐름
1. **그룹 선정**: negative → positive_a → positive_b → mixed → STRONG_BUY 순으로 진행(기존 4개 V2 모듈 경계와 동일).
2. **문구 리라이트**: 그룹에 속한 각 버킷의 base/V1/V3/V4/밈 문구를 V2 톤 가이드(나뚜믄오른다/존버/물타기/깡통/호구/떡상·떡락/개미/뇌동매매/주린이/ㄹㅇ/ㅋㅋ, "10년차 개미 고인물이 후배한테 한마디" 톤)에 맞춰 재작성. 개수·버킷 매핑은 유지, 텍스트만 교체.
3. **자동 검증**: `tests/test_one_liner_consistency.py` 등 기존 규칙(격식체 금지, 기술용어 금지, 숫자·마침표 금지, NEUTRAL 방향성 명령 금지, 중복 금지)을 실행해 위반 문구를 재생성.
4. **표본 육안 검수**: 버킷당 5~10개 무작위 추출해 톤이 실제로 V2급인지 확인.
5. **다음 그룹**: 검증 통과 후 다음 그룹으로 진행.

## 5. 성공 기준

- [ ] 18개 버킷 전체의 base/V1/V3/V4/밈 유래 문구가 V2 수준의 날것 슬랭 톤으로 리라이트된다.
- [ ] STRONG_BUY 버킷이 111개 → 195개(다른 버킷 평균 수준)로 보강된다.
- [ ] 리라이트 후에도 각 버킷의 문구 개수와 버킷-폴라리티(긍정/부정/중립) 매핑이 변경 전과 동일하다.
- [ ] `_bucket()`, `_raw_bucket()`, `get_one_liner()`의 로직(해시 기반 선택, AVOID 강제 전환 등)이 변경되지 않는다 — 문구 텍스트만 교체.
- [ ] 기존 `tests/test_one_liner_consistency.py` 등 관련 테스트가 전부 통과한다(격식체·기술용어·숫자·마침표·중복·NEUTRAL 방향성 규칙).
- [ ] 버킷마다 무작위 표본 5~10개를 육안 검수해 V2 원본 예시(예: "이런 헐값 알짜 발견하면 입꼬리 올라가는 게 본능임", "존버충 인증하면 계좌가 웃어줌" 류)와 동일한 수준의 슬랭 밀도가 확인된다.

## 6. 안 만드는 것 (Out of Scope)

> 이 목록에 있는 건 Phase 1에서 만들지 않습니다.
> AI에게 코드를 시킬 때 이 목록을 함께 공유하세요.

- `_bucket()`, `_raw_bucket()`, `_score_grade()` 등 버킷 분류/등급 로직 변경 -- 이유: 이번 작업은 문구 "내용"만 교체하는 톤 통일 작업, 분류 로직은 이미 검증된 별개 시스템.
- `get_one_liner()`의 해시 기반 선택 알고리즘 변경 -- 이유: 선택 방식이 아니라 선택되는 문구 자체의 톤이 문제.
- base/V1/V2/V3/V4/밈 세대 구분 자체를 단일 파일로 통합하는 리팩터링 -- 이유: Phase 3 후보로 이연(기존 handover 문서의 미완 과제와 별개 트랙).
- `web_app/templates/compare.html` 등 한줄평 외 다른 화면의 문구 톤 변경 -- 이유: 범위를 `one_liner.py` 관련 파일로 한정.
- 실사용 노출 데이터 기반 문구 성과 측정/교체 -- 이유: Phase 2로 이연, 이번 골에는 노출 데이터 수집 인프라가 없음.
- 새 문구 생성 알고리즘(LLM 실시간 호출 등) 도입 -- 이유: 기존처럼 정적 문구 풀 + 해시 선택 방식 유지.
