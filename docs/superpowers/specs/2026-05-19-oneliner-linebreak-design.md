# 한줄평 스마트 줄바꿈 설계

**날짜:** 2026-05-19
**상태:** 승인됨 (구현 대기)

## 문제

상세 패널 포스터(`.hero-oneliner`)의 한줄평이 한국어 구문 경계를 무시하고
어색한 위치에서 줄바꿈된다. 예: "뭐 하나 터져야 움직일 / 텐데 터질 게 없는
종목임" — "움직일 텐데"가 두 줄로 쪼개짐.

근본 원인: CSS `text-wrap: balance`가 한국어 어절/구문 구조를 모른 채 좌우
시각 폭만 맞춰 가장 가까운 공백에서 끊는다. `word-break: keep-all`은 단어
내부 분리만 막을 뿐, 어느 어절 경계에서 끊을지는 제어하지 못한다.

## 목표

- 긴 한줄평이 두 줄이 될 때 **자연스러운 구문 경계**에서 끊는다.
- 한 줄에 들어가는 짧은 문구는 강제로 쪼개지 않는다.
- 큐레이션된 문구 풀(~1000+)을 손대지 않고 자동 적용한다.
- 표시되는 모든 위치에 일관 적용한다.

## 비목표 (YAGNI)

- 문구 풀 수동 줄바꿈 마커 편집.
- 임계값의 환경변수/런타임 설정 (모듈 상수면 충분).
- 비교 뷰만을 위한 별도 분기 (동일 함수가 자동 적용됨).
- 형태소 분석기 등 외부 의존성 도입.

## 아키텍처

`one_liner.py`에 순수 함수 하나를 추가한다:

```
wrap_oneliner(text: str) -> str
```

- 부수효과 없음, 입력 → 출력만. 단위 테스트 용이.
- 한줄평 문구가 최종 확정된 직후(회전/해시 선택 후, 반환 직전) 1회 통과.
- 서버 단일 지점 수정 → 본문을 표시하는 3곳에 자동 반영:
  - 상세 패널 포스터 `.hero-oneliner` (app.js ~955)
  - 하이쿠 패널 (app.js ~1562)
  - 비교 뷰 `.compare-oneliner` (app.js ~2836)
- 스캐너 목록은 `OneLinerTag`(태그)만 사용하고 본문 문장을 렌더하지
  않으므로 영향 없음.
- `haiku_lines.py` 등 별도 문구 소스가 최종 OneLiner를 만드는 경로가
  있으면 동일 함수를 그 반환 지점에도 적용한다 (구현 시 확인).

## 줄바꿈 위치 휴리스틱

입력은 공백으로 구분된 어절 시퀀스. **최대 1곳**의 분리 지점을 고르고
그 자리에 `\n` 1개를 삽입한다 (문자 삽입만, 손실 없음).

1. **길이 게이트:** 공백 제외 글자 수가 임계값(`_WRAP_MIN_CHARS`,
   초기값 13, 포스터 폰트에서 한 줄 ≈ 한글 12자 안팎 기준) 이하면
   원문을 그대로 반환한다.
2. **쉼표 뒤 (최우선):** 문자열에 `,`가 있으면 그 뒤 어절 경계에서
   분리. 여러 개면 문자열 중앙에 가장 가까운 쉼표.
3. **연결/종결 어미 뒤:** 어절이 다음 어미로 끝나면 그 어절 뒤를
   후보로 본다 — `데, 텐데, 는데, ㄴ데, 지만, 니까, 고, 며, 면, 서,
   데도`. 중앙에 가장 가까운 후보 선택.
4. **의존 토큰 앞 분리 금지:** 다음으로 *시작하는* 어절은 줄 첫머리에
   오지 못한다 (앞 어절에 붙임): `텐데, 듯, 뿐, 만, 게, 거, 거임, 줄,
   채, 척, 따름, 양`. 해당 위치의 후보 경계는 제거.
5. **폴백 (균형 분리):** 위에서 확정 못 하면, 유효한 어절 경계 중
   좌우 글자 수 차이가 가장 작은 곳을 고른다. 한쪽이 1어절뿐인
   외톨이 줄은 회피한다.
6. **분리 불가:** 유효 경계가 전혀 없으면(예: 어절 1개) `\n` 없이
   원문 반환. CSS 폴백(`word-break: keep-all` + 일반 wrap)에 맡긴다.

스샷 사례 "뭐 하나 터져야 움직일 텐데 터질 게 없는 종목임":
규칙 4가 "텐데" 앞 분리를 막고, 규칙 3이 "텐데"(종결 어미)를 후보로
잡아 → "뭐 하나 터져야 움직일 텐데 / 터질 게 없는 종목임".

## 렌더링 (XSS 안전)

본문 표시 3곳에서 기존 `esc(d.OneLiner)` **이후**에 `\n` → `<br>`로
치환한다. HTML 이스케이프가 끝난 문자열에 대해서만 치환하므로 주입
위험 없음. 3곳이 공유하는 작은 헬퍼로 구현:

```js
function olHtml(text){ return esc(text).replace(/\n/g, '<br>'); }
```

뒤에 붙는 `OneLinerData` 통계 스팬은 별도 요소이므로 영향 없음.

CSS 변경: `.hero-oneliner`에서 `text-wrap: balance` **제거**(어색한
분리의 직접 원인). `word-break: keep-all`은 좁은 화면에서 한 emit
라인이 또 넘칠 때를 위한 폴백으로 유지.

## 테스트

`tests/` 에 `wrap_oneliner` 단위 테스트:

- 스샷 문구 → "텐데" 뒤에서 분리, "움직일"과 "텐데" 같은 줄.
- 쉼표 포함 문구 → 쉼표 뒤 분리.
- 짧은 문구(임계값 이하) → 무변경.
- 어떤 결과도 의존 토큰(텐데/게/거임 등)으로 줄 시작 안 함.
- `result.replace("\n","")` == 원문 (문자 손실/추가 없음).
- 멱등성: `wrap_oneliner(wrap_oneliner(x)) == wrap_oneliner(x)`.
- 분리 경계 없는 단일 어절 → `\n` 없음.
