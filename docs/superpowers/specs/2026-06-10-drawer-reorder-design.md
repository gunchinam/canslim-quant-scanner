# 드로워(종목 상세 패널) 재구성 설계 — C 하이브리드

작성일: 2026-06-10 · 대상: `web_app/templates/scanner.html` 의 `#detail-panel` (드로워)
상태: 설계 확정 (브레인스토밍 승인 완료) → 구현 계획 대기

---

## 1. 배경 / 문제

종목을 클릭하면 열리는 드로워가 "뒤죽박죽"하게 느껴진다. 진단한 원인 4가지:

1. **Hero(요약) 과부하** — 한 섹션에 한줄평·점수·시그널·리스크·가격·등락·RSI·RS·스파크라인·52주·거래량·시총·뉴스·실적 14가지가 몰림.
2. **"차트" 탭 이름↔내용 불일치** — 탭 이름은 "📈 차트"인데 핸드드로잉 차트가 제거돼 실제론 4축 타이밍 분석만 표시.
3. **"분석" 섹션이 잡화점** — 단발 카드 4개(드로다운·컨센서스·미드캡·투자자수급) + 탭 5개(CANSLIM·기술·재무·공시뉴스·US인사이트)가 혼재.
4. **"수급" 정보 분산** — 4축의 수급 축 · 투자자 카드(기관·외국인) · 기술 탭 세 곳에 흩어짐.
5. **회사 정보가 정반대 위치** — 사용자는 "회사 먼저(뭐 하는 곳·해자)"를 원하는데, 해자·경쟁사가 맨 아래에 **접힌 채** 있음.

## 2. 목표 / 비목표

**목표**
- "회사 먼저(제대로)" 서사: 회사 정체성 → 결론 → 타이밍 → 상세 → 참고.
- 이해하기 쉬운 선형 흐름 + 무거운 콘텐츠는 접어서 첫 화면 부담↓ (C 하이브리드).
- **정보 손실 0** — 모든 기존 요소를 새 위치로 재배치.

**비목표 (out of scope)**
- 점수/등급/시그널 계산 로직 변경 없음 (순수 재배치/그룹핑).
- 신규 데이터·API 없음.
- 비주얼 테마(색·폰트) 대개편 없음 — 순서·그룹핑 중심의 정리.
- `detail.html`(별도 전체 상세 페이지)은 **변경하지 않음** — 드로워(`scanner.html`)만 대상.

## 3. 현재 구조 (요약)

```
헤더(종목명·티커·섹터·외부링크)
dp-about-box (회사 한줄 설명)
dp-section-nav: 📋요약 | 📈차트 | 📊분석 | 🏢기업 | 📎참고
① dp-hero-zone     — 한줄평+점수+시그널+리스크+가격+등락+RSI+RS+스파크+52주+거래량+시총+뉴스+실적
② dp-chart-section — 4축(추세/모멘텀/변동성/수급)+별점+국면+핵심관찰  ("차트"인데 차트 없음)
③ dp-analysis-section — 드로다운·컨센서스·미드캡·투자자수급 + 탭(CANSLIM/기술/재무/공시뉴스/US인사이트)
④ dp-group-company (접힘) — 해자·경쟁사 피어
⑤ dp-group-ref (접힘) — 참고
```

## 4. 새 구조 (C 하이브리드)

```
헤더(고정)
dp-section-nav(이름 변경): 🏢회사 | 🎯결론 | 📈타이밍 | 📊상세 | 📎참고
① 🏢 회사 정체성        — 스크롤 (최상단)
② 🎯 결론               — 스크롤
③ 📈 타이밍             — 스크롤
④ 📊 상세 (접힘)
⑤ 📎 참고 (접힘)
```

### 섹션별 상세 + 요소 매핑 (기존 element ID 보존)

**① 🏢 회사 정체성** ‹↑ 맨 아래에서 최상단으로›
- 회사 한줄 설명(`dp-about`) + 섹터(`dp-sector`) + 시총(`dp-mktcap`)
- 🛡 핵심 해자 (`dp-moat-card` 일체)
- 경쟁사 포지션 (`dp-peers-card`)

**② 🎯 결론** ‹Hero 슬림화 — "판정"만›
- 한줄평 포스터(`dp-fa-haiku`)
- Zone A: 종합점수(`dp-score`) · 시그널(`dp-signal`) · 리스크(`dp-risk-gauge`) · 주도주 배지(`dp-leader-badge`)
- 기존 Hero에서 가격·RSI·스파크·52주·거래량은 ③타이밍으로 이동.

**③ 📈 타이밍** ‹✎ "차트"→"타이밍" 이름·내용 일치 · ⇄ 수급 통합›
- 가격 블록: 현재가(`dp-price`)·등락률(`dp-day-chg`)·스파크라인(`dp-spark-panel`)·52주 위치(`dp-wk52-*`) ‹↓ Hero에서›
- 4축 타이밍: 추세/모멘텀/변동성/수급(`dp-fouraxis-header`) + 진입 별점(`dp-fa-stars`) + 국면(`dp-fa-phase`)
- RSI 온도계(`dp-thermo-*`, `dp-thermo-marker`) ‹↓ Hero에서›
- 💧 **수급 통합**: 거래량비(`dp-volratio`) + 기관·외국인 매매(`dp-investor-card`) + 4축 수급 축을 인접 배치 ‹⇄ 세 곳→한 곳›
- 핵심 관찰(`dp-fouraxis-obs`)

**④ 📊 상세 (접힘)** ‹기존 "분석"의 탭/카드 이전›
- 재무·실적(`dp-earnings-card`, `dp-tab-finance`) · 컨센서스(`dp-consensus-card`) · 미드캡 알파(`dp-midcap-card`) · 드로다운 리스크(`dp-drawdown-risk`)
- CANSLIM 항목별(`dp-tab-canslim`) · 기술 상세(`dp-tab-tech`)
- 공시·뉴스(`dp-tab-dartnews`) · US 인사이트(`dp-tab-usinsight`)
- 분석 탭(`switchDpTab`) 구조는 상세 섹션 내부에 유지.

**⑤ 📎 참고 (접힘)** — 기타 참고(`dp-group-ref`)

## 5. 핵심 변화 4가지

1. **회사 정체성 → 최상단** (해자·경쟁사를 맨 아래 접힘에서 위로).
2. **Hero 분리**: 결론(판정) ↔ 타이밍(가격·기술) 으로 갈라 과부하 해소.
3. **수급 통합**: 세 곳 분산 → 타이밍 한 곳.
4. **"차트"→"타이밍"** 이름·내용 일치 + 무거운 상세는 접기.

## 6. 구현 제약 / 노트

- **element ID 보존 (최우선)**: `app.js`가 ID로 DOM을 조작하므로(`dp-score`, `dp-fouraxis-header`, `dp-investor-card` 등), **ID는 유지하고 DOM 위치(부모 섹션)만 재배치**한다. ID 변경 금지.
- **영향 파일**:
  - `web_app/templates/scanner.html` — 섹션 재배치, nav 라벨/`dpScrollTo` 타겟 갱신, 접힘 그룹(`dp-card-group`) 적용.
  - `web_app/static/scanner.css` — 섹션 배경 교차/간격 등 새 순서에 맞춤.
  - `web_app/static/app.js` — 섹션 nav `dpScrollTo` 타겟 id, `switchDpTab`(상세 섹션 내 유지), 데이터 채우는 로직은 ID 기반이라 대부분 불변.
- **접힘 패턴 재사용**: 기존 `dp-card-group` + `toggleCardGroup(this)` 패턴으로 ④상세·⑤참고 구현.
- **수급 통합**: `dp-investor-card`를 타이밍 섹션으로 이동하고 거래량비·4축 수급과 인접 배치(중복 라벨 정리).
- **로딩/이름**: "분석 불러오는 중" 로딩 메시지는 ③타이밍 섹션에 귀속(이미 "차트 생성 중"→"분석 불러오는 중" 수정됨).

## 7. 미해결 / 후속 판단

- 모바일에서 ④상세·⑤참고 접힘 기본값 유지(권장) — 데스크톱도 동일.
- 수급 통합 카드의 정확한 레이아웃(거래량비 + 기관/외국인 + 4축 수급을 한 카드로 합칠지, 인접 배치만 할지)은 구현 계획 단계에서 확정.
- ③타이밍 내부 순서(가격 블록 먼저 vs 4축 먼저)는 구현 시 미세 조정.
